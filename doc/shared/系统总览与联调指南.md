# 系统总览与联调指南

这份文档把仓库里几个子项目之间的关系、当前真正可运行的链路，以及联调时应该怎么分步验证，尽量用一页讲清楚。

## 一句话理解

当前系统的核心是：

`服务端 ESP32-S3 负责采集、自动化和控制；浏览器和客户端只是它的控制入口；ESP8266 红外板是它未来的外设执行端。`

## 角色划分

### `esp32_home_server`

职责：

- 采集 DHT11、光照、MQ2、火焰传感器
- 运行自动化规则
- 提供本地网页控制台
- 提供 HTTP 状态和控制接口
- 控制风扇、窗帘、蜂鸣器
- 把红外字符串命令转发到 ESP8266

现状：

- 是仓库里当前最成熟、最应优先理解的工程

### `esp32_home_client`

职责：

- 作为第二控制入口
- 通过 HTTP 拉服务端状态
- 通过 HTTP 发控制命令
- 负责本地按键、旋钮、RGB 和 LCD 交互

现状：

- 可用于联调和理解控制面板思路
- 个别文档表述还没有完全跟上服务端最近的协议收敛

### `esp8266_ir_control`

职责：

- 接收服务端经 UART 发来的字符串命令
- 将字符串映射为实际红外发射动作

现状：

- 目录已建立
- 当前代码仍是默认骨架，还没有成为真正可用的红外执行端

### `esp32_ir_Study`

职责：

- 用于红外协议学习和实验

现状：

- 不在当前主链路上

## 当前主链路

### 链路 A：浏览器 -> 服务端

用途：

- 本地网页调试
- 状态观察
- 直接控制风扇和窗帘

接口：

- `GET /api/status`
- `POST /api/control`

### 链路 B：客户端 -> 服务端

用途：

- 作为专用局域网控制面板

接口：

- 同样走 `/api/status` 和 `/api/control`

### 链路 C：服务端 -> ESP8266 红外桥

用途：

- 将红外操作抽象为字符串命令

示例命令：

- `ac_power`
- `ac_temp_up`
- `ac_temp_down`
- `ac_mode`

说明：

- 服务端不关心红外编码细节
- 最终发码逻辑应由 ESP8266 端承担

## 当前网络模式

### `cloud`

- 服务端成功接入 `iPadmini`
- 本地 HTTP 仍可用
- mDNS 可用

### `local_ap`

- 服务端未连上外部 Wi-Fi
- 自动开启热点 `esp32-server`
- 浏览器和客户端仍可直接访问服务端 HTTP 接口

## 当前已经收敛的协议约定

- 主控制协议是 HTTP，不是 MQTT
- 红外控制是字符串命令，不是服务端直接发红外码
- 风扇控制已经收敛为 `mode` 或 `speedPercent`
- 窗帘控制使用 `angle` 或 `preset`

## 推荐联调顺序

1. 单独验证服务端启动、联网、网页和 HTTP 接口
2. 单独验证风扇驱动链路
3. 单独验证窗帘舵机链路
4. 单独验证传感器采样
5. 在服务端稳定后再接入客户端
6. 最后单独实现并接入 ESP8266 红外桥

## 近期实现重点

从最近几次提交可以看出，项目重点已经偏向“让服务端在现场稳定可用”：

- 自动化规则补全
- 本地网页控制台完善
- 风扇 PWM 调速收敛
- 控制延迟优化
- 执行器阻塞问题修复
- 舵机干扰和卡顿风险缓解

这意味着你现在阅读代码时，应该优先相信服务端当前实现，而不是历史设想。

## 常见理解误区

- 看到 MQTT 代码，不代表 MQTT 已经是主链路
- 看到 `esp8266_ir_control/` 目录，不代表红外桥已经实现
- 看到客户端文档中的旧按钮描述，不代表服务端还保留同样的控制语义
- 看到实验工程，不代表它参与当前运行链路
