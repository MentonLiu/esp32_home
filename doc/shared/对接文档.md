# ESP32 客户端与 MQTT 上位机对接文档

本文档当前主要服务于“ESP32 局域网客户端”和“服务端 HTTP 接口”的联调。MQTT 还没有定下最终方案，因此这里只保留最小的占位说明，不把它当成当前主链路。

## 1. 当前通信分工

### 1.1 ESP32 局域网客户端

ESP32 客户端通过局域网直接访问服务端 HTTP 接口，适合做：

- 本地按钮/旋钮控制器
- 小型显示终端
- 第二块 ESP32 控制面板



## 2. 服务端运行模式

服务端运行模式由 `ConnectivityManager` 管理。

### 2.1 `cloud` 模式

- 服务端连接到家庭 Wi-Fi
- 本地 HTTP 接口可继续使用
- MQTT 代码可保留，但当前不作为主功能
- `GET /api/status` 返回 `mode = "cloud"`

当前源码内置的站点 Wi-Fi 配置在 [CentralProcessor.cpp](../../esp32_home_server/src/CentralProcessor.cpp)：

- SSID：`iPadmini`
- Password：`lbl450981`

### 2.2 `local_ap` 模式

- 服务端无法连上外网时自动进入该模式
- 服务端开启热点，供局域网客户端接入
- HTTP 接口可用
- MQTT 不作为当前能力保证
- `GET /api/status` 返回 `mode = "local_ap"`

当前热点配置同样在 [CentralProcessor.cpp](../../esp32_home_server/src/CentralProcessor.cpp)：

- SSID：`esp32-server`
- Password：`lbl450981`

补充说明：

- 服务端每 30 秒重新检查一次联网状态
- 有网时切回 `cloud`
- 无网时切回 `local_ap`

## 3. ESP32 局域网客户端对接

### 3.1 通信前提

客户端和服务端必须在同一个局域网内。这个局域网可能是：

1. 家用路由器 Wi-Fi
2. 服务端自己开启的 `esp32-server` 热点

推荐流程：

1. 客户端连入与服务端相同的网络
2. 获取服务端当前 IP
3. 通过 HTTP 访问 `/api/status`
4. 通过 HTTP 向 `/api/control` 发送命令

如果是服务端热点场景，通常可以先尝试：

- `http://192.168.4.1/api/status`

### 3.2 本地 HTTP 接口归属

本地接口由 [LocalProcessingProgram.cpp](../../esp32_home_server/src/LocalProcessingProgram.cpp) 提供。

当前已实现的核心路由有：

- `GET /`
- `GET /api/status`
- `POST /api/control`

ESP32 客户端真正需要的是后两个接口。

### 3.3 获取状态

请求：

```http
GET /api/status HTTP/1.1
Host: <server-ip>
```

返回示例：

```json
{
  "mode": "local_ap",
  "ip": "192.168.4.1",
  "temperatureC": 26.3,
  "humidityPercent": 58.0,
  "lightPercent": 72,
  "mq2Percent": 18,
  "smokeLevel": "green",
  "flameDetected": false,
  "fanMode": "off",
  "fanPowerOn": false,
  "fanSpeedPercent": 0,
  "curtainAngle": 90,
  "error": false,
  "errorMessage": ""
}
```

字段说明：

- `mode`：服务端运行模式，`cloud` 或 `local_ap`
- `ip`：服务端当前局域网 IP
- `temperatureC`：温度，单位摄氏度
- `humidityPercent`：湿度百分比
- `lightPercent`：光照百分比，范围 `0-100`
- `mq2Percent`：MQ2 百分比，范围 `0-100`
- `smokeLevel`：`green`、`blue`、`yellow`、`red`
- `flameDetected`：是否检测到火焰
- `fanMode`：风扇档位，`off`、`low`、`medium`、`high`
- `fanPowerOn`：风扇电源状态，只有为 `true` 时才允许调速
- `fanSpeedPercent`：风扇当前速度百分比
- `curtainAngle`：窗帘当前角度，范围 `0-180`
- `error`：采样是否异常
- `errorMessage`：错误信息，常见值为 `dht_read_failed`

### 3.4 发送控制命令

请求：

```http
POST /api/control HTTP/1.1
Host: <server-ip>
Content-Type: application/json
```

返回：

```json
{
  "ok": true,
  "stateChanged": true,
  "type": "local_command",
  "message": "fan_updated"
}
```

注意：

- 当前 HTTP 会返回真实命令执行结果
- 命令 JSON 非法时会返回 `400`
- 真正的命令解析结果由 `ControllerCommandProcessor` 决定
- 客户端发命令后，仍建议重新拉一次 `/api/status` 做状态确认

### 3.5 局域网控制命令格式

所有命令都由 [ControllerCommandProcessor.cpp](../../esp32_home_server/src/ControllerCommandProcessor.cpp) 解析。

公共字段：

- `device`

支持的设备：

- `fan`
- `curtain`
- `ir`

#### 风扇

电源控制：

```json
{"device":"fan","power":"on"}
```

```json
{"device":"fan","power":"off"}
```

按档位：

```json
{"device":"fan","mode":"off"}
```

```json
{"device":"fan","mode":"low"}
```

```json
{"device":"fan","mode":"medium"}
```

```json
{"device":"fan","mode":"high"}
```

按百分比：

```json
{"device":"fan","speedPercent":80}
```

说明：

- `power` 允许 `on`、`off`
- `mode` 允许 `off`、`low`、`medium`、`high`
- `speedPercent` 建议范围 `0-100`
- 风扇只有在 `power=on` 时才允许调速
- 如果 `power=off`，服务端会强制把转速归零
- 如果同时传 `mode` 和 `speedPercent`，最终以 `speedPercent` 覆盖结果为准

#### 窗帘

按角度：

```json
{"device":"curtain","angle":135}
```

按预设：

```json
{"device":"curtain","preset":2}
```

预设映射：

- `0 -> 0°`
- `1 -> 45°`
- `2 -> 90°`
- `3 -> 135°`
- `4 -> 180°`

说明：

- `angle` 建议范围 `0-180`
- 如果同时传 `angle` 和 `preset`，最终以 `preset` 对应角度为准

#### 红外桥接

服务端不会直接发红外，而是通过 UART 向外接 ESP8266 发送字符串命令。

示例：

```json
{"device":"ir","command":"ac_power"}
```

```json
{"device":"ir","command":"ac_temp_up"}
```

```json
{"device":"ir","command":"ac_temp_down"}
```

```json
{"device":"ir","command":"ac_mode"}
```

说明：

- `command` 是给 ESP8266 的字符串命令
- ESP8266 端自行解释这个字符串并执行对应红外协议
- 当前不再把红外协议细节放在服务端和客户端里

### 3.6 轮询与刷新建议

建议 ESP32 客户端按以下节奏工作：

1. 每 `500ms ~ 1000ms` 拉取一次 `/api/status`
2. 用户下发命令后等待 `200ms ~ 500ms`
3. 再次拉取 `/api/status`
4. 用新状态更新本地显示

### 3.7 局域网客户端注意点

1. 客户端不要把 MQTT 当作本地 ESP32 控制链路的必需条件。
2. 模式切换时服务端 IP 可能变化，不要长期写死。
3. `mode=local_ap` 时，HTTP 仍可用。
4. 客户端现在不再依赖服务端时间字段。
5. 8pin TFT 未来会改成 LVGL，但当前显示内容先不实现。

## 5. 自动化行为

自动化由 [AutomationEngine.cpp](../../esp32_home_server/src/AutomationEngine.cpp) 负责。

当前已实现：

- 每天 `07:00` 自动打开窗帘
- 每天 `22:00` 自动关闭窗帘
- `mq2Percent >= 75` 时自动风扇高速
- `mq2Percent >= 90` 时蜂鸣器短鸣
- 火焰持续 `45s` 后播放“短短长”
- 火焰持续 `5min` 时保留火警占位处理逻辑

## 6. 联调建议

### 6.1 ESP32 局域网客户端联调

建议顺序：

1. 让客户端和服务端进入同一个局域网
2. 浏览器访问 `http://<server-ip>/api/status`
3. 用 HTTP 工具向 `/api/control` 发送风扇、窗帘、红外命令
4. 再开始联调客户端按钮、旋钮、RGB、LCD1602

### 6.2 MQTT 上位机联调

当前暂缓，不作为本轮开发目标。

## 7. 当前源码下最值得注意的点

1. HTTP 是局域网 ESP32 客户端的主通道。
2. MQTT 仍未定稿，当前只保留占位代码，不应当影响双端主链路。
3. HTTP 控制接口现在会把命令执行结果直接返回给客户端。
4. 红外链路已经收敛为“字符串命令 -> ESP8266”。
5. 风扇现在需要区分电源状态和转速状态，客户端要先开机再调速。
