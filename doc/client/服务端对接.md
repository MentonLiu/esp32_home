# ESP32 Home Client 服务端对接说明

本文面向 `esp32_home_client` 与 `esp32_home_server` 的当前联调，目标是说明最新服务端协议、连接方式，以及当前客户端与服务端之间仍存在的兼容性差异。

## 1. 对接目标与权威来源

本说明以以下内容为权威来源，冲突时按此顺序判断：

1. `../esp32_home_server/src/SensorDataProcessor.cpp`
2. `../esp32_home_server/src/ControllerCommandProcessor.cpp`
3. `../esp32_home_server/src/LocalProcessingProgram.cpp`
4. `../esp32_home_server/src/ConnectivityManager.cpp`
5. `../esp32_home_server/README.md`
6. `../esp32_home_server/changes.md`

说明：

- `../doc/server/功能要求.md` 属于历史需求文档，可作为背景参考，但不作为当前协议权威定义。
- 当前服务端主链路是本地 HTTP。
- MQTT 仍有保留代码和说明文档，但不是当前客户端联调前提。
- 当前实现中没有单独的“服务端版本号接口”或“协议协商接口”，联调时应直接以服务端当前代码和接口返回结构为准。

## 2. 发现与连接方式

客户端当前按以下顺序寻找服务端：

1. 优先连接家庭 Wi-Fi `HW-2103`
2. 家庭 Wi-Fi 不可用时，回退连接服务端热点 `esp32-server`

服务端存在两种访问模式：

- `cloud`
  服务端已连接家庭路由器，可通过实际局域网 IP 访问，也可尝试 `http://esp32-home-server.local/`
- `local_ap`
  服务端未连上家庭路由器，自动开启热点，客户端可直接访问 `http://192.168.4.1`

当前客户端发现服务端的方式与上述模式对应：

- 连接到服务端热点时，直接探测 `http://192.168.4.1`
- 连接到家庭 Wi-Fi 时，优先通过 mDNS 查询 `esp32-home-server.local`
- mDNS 不可用时，客户端当前没有额外的目录服务或版本协商能力

## 3. 状态接口

### 3.1 `GET /api/status`

用途：返回系统完整状态，供客户端轮询展示和同步控制目标值。

当前返回对象采用“双轨兼容结构”：

- 一份顶层平铺字段
- 一份分组字段 `sensor` 与 `controller`

### 3.2 顶层字段

当前至少包含以下字段：

| 字段              | 类型    | 说明                                |
| ----------------- | ------- | ----------------------------------- |
| `mode`            | string  | `cloud` 或 `local_ap`               |
| `ip`              | string  | 服务端当前访问 IP                   |
| `sensorTimestamp` | number  | 传感器采样时间戳，来源于 `millis()` |
| `temperatureC`    | number  | 温度，单位摄氏度                    |
| `humidityPercent` | number  | 湿度百分比                          |
| `lightPercent`    | number  | 光照百分比                          |
| `mq2Percent`      | number  | MQ2 百分比                          |
| `rainPercent`     | number  | 雨量百分比                          |
| `isRaining`       | boolean | 是否处于下雨状态                    |
| `smokeLevel`      | string  | `green` / `blue` / `yellow` / `red` |
| `fanMode`         | string  | `off` / `low` / `medium` / `high`   |
| `fanSpeedPercent` | number  | 风扇速度百分比                      |
| `curtainAngle`    | number  | 窗帘角度，范围 `0-180`              |
| `error`           | boolean | 采样链路是否有错误                  |
| `errorMessage`    | string  | 错误信息                            |

注意：

- 当前服务端不再输出 `fanPowerOn`
- 当前服务端状态以 `fanMode` 和 `fanSpeedPercent` 表示风扇状态

### 3.3 分组字段

`sensor` 子对象当前至少包含：

- `temperatureC`
- `humidityPercent`
- `lightPercent`
- `mq2Percent`
- `rainPercent`
- `isRaining`
- `smokeLevel`
- `timestamp`
- `error`
- `errorMessage`

`controller` 子对象当前至少包含：

- `fanMode`
- `fanSpeedPercent`
- `curtainAngle`

可选字段：

- `controller.curtainPreset`
  仅当当前控制器状态存在有效预设时返回

### 3.4 示例响应

```json
{
  "mode": "local_ap",
  "ip": "192.168.4.1",
  "sensorTimestamp": 1234567,
  "temperatureC": 26.3,
  "humidityPercent": 58.0,
  "lightPercent": 72,
  "mq2Percent": 18,
  "rainPercent": 5,
  "isRaining": false,
  "smokeLevel": "green",
  "fanMode": "off",
  "fanSpeedPercent": 0,
  "curtainAngle": 90,
  "error": false,
  "errorMessage": "",
  "sensor": {
    "temperatureC": 26.3,
    "humidityPercent": 58.0,
    "lightPercent": 72,
    "mq2Percent": 18,
    "rainPercent": 5,
    "isRaining": false,
    "smokeLevel": "green",
    "timestamp": 1234567,
    "error": false,
    "errorMessage": ""
  },
  "controller": {
    "fanMode": "off",
    "fanSpeedPercent": 0,
    "curtainAngle": 90
  }
}
```

## 4. 控制接口

### 4.1 `POST /api/control`

用途：接收 JSON 控制命令，当前支持 `fan` 和 `curtain` 两类设备。

请求头建议：

```http
Content-Type: application/json
```

### 4.2 风扇控制

按档位控制：

```json
{"device":"fan","mode":"off"}
```

```json
{"device":"fan","mode":"low"}
```

```json
{"device":"fan","mode":"medium"}
```

```json
{"device":"fan","mode":"high"}
```

按速度百分比控制：

```json
{"device":"fan","speedPercent":80}
```

说明：

- `mode` 与 `speedPercent` 任意一种存在即可执行
- 当前协议中不存在 `power` 字段

### 4.3 窗帘控制

按角度控制：

```json
{"device":"curtain","angle":135}
```

按预设控制：

```json
{"device":"curtain","preset":3}
```

说明：

- `angle` 与 `preset` 任意一种存在即可执行
- `preset` 当前由服务端约束在 `0-4`

### 4.4 统一响应结构

成功或失败都返回统一 JSON 结构：

| 字段           | 类型    | 说明               |
| -------------- | ------- | ------------------ |
| `ok`           | boolean | 是否被接收并执行   |
| `stateChanged` | boolean | 是否引起状态变化   |
| `type`         | string  | 来源类型或错误类型 |
| `message`      | string  | 结果消息           |

成功示例：

```json
{
  "ok": true,
  "stateChanged": true,
  "type": "local_command",
  "message": "fan_updated"
}
```

失败时 HTTP 状态码为 `400`，响应体结构不变。

## 5. 错误消息与联调判断

当前服务端会返回的典型失败消息如下：

- `invalid_control_json:*`
- `device_missing`
- `unknown_device`
- `fan_mode_invalid`
- `fan_command_missing`
- `curtain_command_missing`

联调建议：

- 收到 `invalid_control_json:*` 时，优先检查请求体是否为合法 JSON
- 收到 `device_missing` 时，确认是否提供了 `device`
- 收到 `fan_command_missing` 时，确认是否至少传入了 `mode` 或 `speedPercent`
- 收到 `curtain_command_missing` 时，确认是否至少传入了 `angle` 或 `preset`
- 若客户端仍发送 `power` 语义，服务端当前不会识别为有效 fan 命令

## 6. 当前客户端兼容性结论

当前 `esp32_home_client` 与最新服务端协议已完成主链路对齐。

### 6.1 已对齐项

- 风扇控制发送 `mode`（`off/low/medium/high`）
- 窗帘控制发送 `angle`
- 状态解析使用 `fanMode`、`fanSpeedPercent`、`curtainAngle`
- 服务端发现与连接流程和当前网络模式一致（家庭 Wi-Fi + 热点回退）

### 6.2 已移除的旧语义

- 客户端当前不再发送 `{"device":"fan","power":"on|off"}`
- 客户端当前不再依赖 `fanPowerOn` 字段

### 6.3 仍可继续增强的点

- 客户端当前主要消费平铺字段，对 `controller.curtainPreset` 仅作为可选信息。
- 客户端输入为按键循环策略，尚未提供网页端同等粒度的可视化引导文案。

## 7. 联调自检建议

建议按以下顺序完成联调验收：

1. 访问 `GET /api/status`，确认 `mode`、`ip`、`rainPercent`、`isRaining` 字段存在
2. 使用客户端按钮 1，确认收到 `{"device":"curtain","angle":...}` 并返回 `curtain_updated`
3. 使用客户端按钮 2，确认收到 `{"device":"fan","mode":"..."}` 并返回 `fan_updated`
4. 断开家庭 Wi-Fi，确认客户端可回退连接 `esp32-server` 并继续控制

## 8. 后续优化建议

1. 在客户端页面文案补充 `rainPercent/isRaining` 的含义说明
2. 为按钮 3/4 定义实际业务能力，减少保留键长期闲置
3. 对 mDNS 发现失败场景补充额外诊断提示
