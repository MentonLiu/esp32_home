# ESP32 Home Client 服务端对接说明

本文面向 `esp32_home_client` 与 `esp32_home_server` 的当前联调，目标是说明最新服务端协议、连接方式，以及当前客户端与服务端之间仍存在的兼容性差异。

## 1. 对接目标与权威来源

本说明以以下内容为权威来源，冲突时按此顺序判断：

1. `../esp32_home_server/src/SensorDataProcessor.cpp`
2. `../esp32_home_server/src/ControllerCommandProcessor.cpp`
3. `../esp32_home_server/src/LocalProcessingProgram.cpp`
4. `../esp32_home_server/src/ConnectivityManager.cpp`
5. `../esp32_home_server/README.md`
6. `../esp32_home_server/changes.md`

说明：

- `../doc/server/功能要求.md` 属于历史需求文档，可作为背景参考，但不作为当前协议权威定义。
- 当前服务端主链路是本地 HTTP。
- MQTT 仍有保留代码和说明文档，但不是当前客户端联调前提。
- 当前实现中没有单独的“服务端版本号接口”或“协议协商接口”，联调时应直接以服务端当前代码和接口返回结构为准。

## 2. 发现与连接方式

客户端当前按以下顺序寻找服务端：

1. 优先连接家庭 Wi-Fi `iPadmini`
2. 家庭 Wi-Fi 不可用时，回退连接服务端热点 `esp32-server`

服务端存在两种访问模式：

- `cloud`
  服务端已连接家庭路由器，可通过实际局域网 IP 访问，也可尝试 `http://esp32-home-server.local/`
- `local_ap`
  服务端未连上家庭路由器，自动开启热点，客户端可直接访问 `http://192.168.4.1`

当前客户端发现服务端的方式与上述模式对应：

- 连接到服务端热点时，直接探测 `http://192.168.4.1`
- 连接到家庭 Wi-Fi 时，优先通过 mDNS 查询 `esp32-home-server.local`
- mDNS 不可用时，客户端当前没有额外的目录服务或版本协商能力

## 3. 状态接口

### 3.1 `GET /api/status`

用途：返回系统完整状态，供客户端轮询展示和同步控制目标值。

当前返回对象采用“双轨兼容结构”：

- 一份顶层平铺字段
- 一份分组字段 `sensor` 与 `controller`

### 3.2 顶层字段

当前至少包含以下字段：

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `mode` | string | `cloud` 或 `local_ap` |
| `ip` | string | 服务端当前访问 IP |
| `sensorTimestamp` | number | 传感器采样时间戳，来源于 `millis()` |
| `temperatureC` | number | 温度，单位摄氏度 |
| `humidityPercent` | number | 湿度百分比 |
| `lightPercent` | number | 光照百分比 |
| `mq2Percent` | number | MQ2 百分比 |
| `rainPercent` | number | 雨量百分比 |
| `isRaining` | boolean | 是否处于下雨状态 |
| `smokeLevel` | string | `green` / `blue` / `yellow` / `red` |
| `fanMode` | string | `off` / `low` / `medium` / `high` |
| `fanSpeedPercent` | number | 风扇速度百分比 |
| `curtainAngle` | number | 窗帘角度，范围 `0-180` |
| `error` | boolean | 采样链路是否有错误 |
| `errorMessage` | string | 错误信息 |

注意：

- 当前服务端不再输出 `fanPowerOn`
- 当前服务端状态以 `fanMode` 和 `fanSpeedPercent` 表示风扇状态

### 3.3 分组字段

`sensor` 子对象当前至少包含：

- `temperatureC`
- `humidityPercent`
- `lightPercent`
- `mq2Percent`
- `rainPercent`
- `isRaining`
- `smokeLevel`
- `timestamp`
- `error`
- `errorMessage`

`controller` 子对象当前至少包含：

- `fanMode`
- `fanSpeedPercent`
- `curtainAngle`

可选字段：

- `controller.curtainPreset`
  仅当当前控制器状态存在有效预设时返回

### 3.4 示例响应

```json
{
  "mode": "local_ap",
  "ip": "192.168.4.1",
  "sensorTimestamp": 1234567,
  "temperatureC": 26.3,
  "humidityPercent": 58.0,
  "lightPercent": 72,
  "mq2Percent": 18,
  "rainPercent": 5,
  "isRaining": false,
  "smokeLevel": "green",
  "fanMode": "off",
  "fanSpeedPercent": 0,
  "curtainAngle": 90,
  "error": false,
  "errorMessage": "",
  "sensor": {
    "temperatureC": 26.3,
    "humidityPercent": 58.0,
    "lightPercent": 72,
    "mq2Percent": 18,
    "rainPercent": 5,
    "isRaining": false,
    "smokeLevel": "green",
    "timestamp": 1234567,
    "error": false,
    "errorMessage": ""
  },
  "controller": {
    "fanMode": "off",
    "fanSpeedPercent": 0,
    "curtainAngle": 90
  }
}
```

## 4. 控制接口

### 4.1 `POST /api/control`

用途：接收 JSON 控制命令，当前支持 `fan` 和 `curtain` 两类设备。

请求头建议：

```http
Content-Type: application/json
```

### 4.2 风扇控制

按档位控制：

```json
{"device":"fan","mode":"off"}
```

```json
{"device":"fan","mode":"low"}
```

```json
{"device":"fan","mode":"medium"}
```

```json
{"device":"fan","mode":"high"}
```

按速度百分比控制：

```json
{"device":"fan","speedPercent":80}
```

说明：

- `mode` 与 `speedPercent` 任意一种存在即可执行
- 当前协议中不存在 `power` 字段

### 4.3 窗帘控制

按角度控制：

```json
{"device":"curtain","angle":135}
```

按预设控制：

```json
{"device":"curtain","preset":3}
```

说明：

- `angle` 与 `preset` 任意一种存在即可执行
- `preset` 当前由服务端约束在 `0-4`

### 4.4 统一响应结构

成功或失败都返回统一 JSON 结构：

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `ok` | boolean | 是否被接收并执行 |
| `stateChanged` | boolean | 是否引起状态变化 |
| `type` | string | 来源类型或错误类型 |
| `message` | string | 结果消息 |

成功示例：

```json
{
  "ok": true,
  "stateChanged": true,
  "type": "local_command",
  "message": "fan_updated"
}
```

失败时 HTTP 状态码为 `400`，响应体结构不变。

## 5. 错误消息与联调判断

当前服务端会返回的典型失败消息如下：

- `invalid_control_json:*`
- `device_missing`
- `unknown_device`
- `fan_mode_invalid`
- `fan_command_missing`
- `curtain_command_missing`

联调建议：

- 收到 `invalid_control_json:*` 时，优先检查请求体是否为合法 JSON
- 收到 `device_missing` 时，确认是否提供了 `device`
- 收到 `fan_command_missing` 时，确认是否至少传入了 `mode` 或 `speedPercent`
- 收到 `curtain_command_missing` 时，确认是否至少传入了 `angle` 或 `preset`
- 若客户端仍发送 `power` 语义，服务端当前不会识别为有效 fan 命令

## 6. 当前客户端兼容性差异

当前 `esp32_home_client` 与最新服务端协议并非完全一致，联调时需要特别注意以下差异。

### 6.1 风扇电源语义已不兼容

当前客户端代码仍保留独立风扇电源按钮，并发送：

```json
{"device":"fan","power":"on"}
```

或：

```json
{"device":"fan","power":"off"}
```

但最新服务端已删除该协议分支，当前仅支持：

- `mode`
- `speedPercent`

因此：

- 客户端风扇独立电源按钮与最新服务端不兼容
- 文档与联调说明中不应再把 `power` 作为当前有效协议

### 6.2 客户端仍依赖已删除状态字段

当前客户端 `ServerStatus` 和交互逻辑仍依赖：

- `fanPowerOn`

但最新服务端已经不再返回该字段，因此客户端当前存在以下风险：

- 风扇按钮状态同步不准确
- 编码器进入风扇模式后，可能仍按旧电源开关语义阻断调速

### 6.3 新增状态字段尚未形成明确客户端语义

服务端当前已增加或明确以下字段：

- `rainPercent`
- `isRaining`
- `controller.curtainPreset`

当前客户端尚未对这些字段形成明确的展示或控制语义，文档中应如实说明为“服务端已提供、客户端暂未完整消费”。

### 6.4 当前联调结论

截至当前代码状态，可得出以下结论：

- 窗帘角度控制可对接
- 风扇速度百分比控制可对接
- 状态轮询主链路可对接
- 风扇独立电源按钮语义与最新服务端不兼容

因此当前整体状态应描述为：

> 当前客户端与最新服务端部分联通，但风扇电源语义不兼容。

## 7. 联调自检建议

建议按以下顺序完成联调验收：

1. 访问 `GET /api/status`，确认能看到 `rainPercent`、`isRaining`，且没有 `fanPowerOn`
2. 发送 `{"device":"fan","mode":"low"}`，确认服务端返回 `ok: true`
3. 发送 `{"device":"fan","speedPercent":80}`，确认服务端返回 `fan_updated`
4. 发送 `{"device":"curtain","angle":135}`，确认服务端返回 `curtain_updated`
5. 发送 `{"device":"curtain","preset":3}`，确认服务端返回 `curtain_updated`
6. 使用当前客户端测试风扇独立电源按钮，预期会暴露旧协议不兼容问题

## 8. 后续迁移优先级

如果后续进入客户端代码迁移阶段，第一优先级应为：

1. 移除 `fanPowerOn` 状态依赖
2. 移除 `{"device":"fan","power":"on|off"}` 发送逻辑
3. 将风扇交互统一收敛为 `mode` / `speedPercent`
